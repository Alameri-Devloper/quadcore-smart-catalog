# ADR-0008: Dynamic Context-Aware Template Generation Engine

## English

## Title

Dynamic Context-Aware Template Generation Engine

## Status

Accepted; implementation is planned for a future task.

## Context

Bulk Product entry must be fast without weakening Product Data Quality. Fixed Excel files maintained manually for every Category, Device Class, Brand, or Product Model would duplicate Catalog rules, become stale when Specification Templates change, and ask employees to repeat information already known from their current Catalog location.

QSC already resolves Product specifications from Category and optional Device Class and supports Context-Aware Product Entry. Template generation must extend those decisions rather than create a separate source of specification or validation rules.

## Decision

QSC must generate import, export, and bulk-operation templates dynamically from the current Entry Context. Templates must not be manually maintained as fixed files for each Catalog combination.

Entry Context may contain:

- `companyId`.
- `workspaceId`.
- `departmentId`.
- `categoryId`.
- `deviceClassId`.
- `brandId`.
- `productModelId`.
- `entryMethod`.
- `language`.
- Future permissions.

The accepted flow is:

```text
Current Catalog Context
-> Specification Template Resolution
-> Known Values Detection
-> Required Column Calculation
-> Dynamic Template Generation
-> Download / Import / Export / Bulk Update
```

Known values are represented in template identity and instructions but are not repeated as editable Product columns. A higher-level context leaves more values unresolved and therefore produces more columns.

## Reasons

- Reduce repeated employee input.
- Prevent mismatched Category, Device Class, Brand, and Product data.
- Keep generated columns synchronized with confirmed Catalog configuration.
- Support high-speed bulk entry without bypassing validation.
- Avoid maintaining many stale fixed files.
- Reuse one engine across import, export, and bulk operations.
- Preserve tenant, workspace, language, and future permission context.

## Consequences

- Template generation requires explicit, confirmed Entry Context.
- Category and optional Device Class continue to resolve the Specification Template under ADR-0002.
- Deeper context produces fewer editable columns; higher context produces more required context columns.
- Newly generated templates reflect Specification Template changes automatically.
- Previously generated templates may require version and context validation during import.
- Context-provided fields must remain visible in identity and preview metadata even when excluded from Product rows.
- Template generation, workbook formatting, storage, download, upload, and parsing require separate implementation design.
- This ADR does not implement Excel, CSV, import, export, or bulk update.

## Architecture Direction

The Template Generation Engine is an application/domain service capability, not UI or Excel-specific business logic.

```text
Product Entry / Export / Bulk UI
-> Template Generation Service
-> Catalog Context Resolution
-> Specification Template Service
-> Product Field Requirements
-> Template Definition
-> Format Adapter (Excel / CSV / future API)
```

Components request previews and generated artifacts through services. Services calculate known, required, optional, included, and excluded fields. Repositories provide confirmed Catalog configuration. Format adapters translate one format-neutral Template Definition into Excel, CSV, or future exchange formats.

Template columns must come only from:

- Confirmed Catalog context.
- The resolved Specification Template.
- Specification Template Fields.
- Confirmed Product field requirements.

Specification definitions must not be duplicated in Excel-specific code or documents. Separate use cases must not create separate template calculation logic.

## Template Identity

Every generated template must identify its owner, context, language, and definition version. A visible header must include applicable values from:

- Department.
- Category.
- Device Class.
- Brand.
- Product Model.
- Company or Workspace.
- Generated By.
- Generated At.
- Template Version.
- Template ID.
- Language.

Example:

```text
QSC Product Template
Department: Computers
Category: Laptops
Device Class: Gaming
Brand: Lenovo
Workspace: Quadcore
Template ID: TMP-LAP-GAM-LEN-001
Template Version: 1.0
Generated At: 2026-07-15
```

File names must be meaningful, such as:

```text
QSC_Computers_Laptops_Gaming_Lenovo_Template_v1.xlsx
QSC_Laptops_Business_Dell_2026-07-15.xlsx
```

Generic names such as `Template.xlsx` are not permitted.

## Self-Describing Template Rules

Every generated template must be understandable without external documentation. A future workbook should contain:

1. Products sheet.
2. Instructions sheet.
3. Optional Reference Data sheet.

The Instructions sheet should explain:

- Template purpose.
- Current context.
- Required and optional fields.
- Context-provided and excluded fields.
- Allowed values.
- Examples.
- Common mistakes.
- Import instructions.
- Template version.

Required, optional, context-provided, and excluded fields must be clearly distinguished. Required and optional rules come from the resolved Specification Template and Product requirements.

Future smart features may include required-field indicators, drop-down lists, allowed-value rules, number and Boolean validation, example values, field descriptions, multiple languages, and a mobile-friendly preview.

Before download, the preview should show current context, included columns, excluded columns, required fields, optional fields, and the reason for every exclusion.

## Import Validation

On upload, QSC must read and validate Template ID, Template Version, Department, Category, Device Class, Brand, and Workspace when applicable. It must compare template identity with the current import context and must not continue silently after a mismatch.

A mismatch message must explain:

- The context the file belongs to.
- The employee's current import context.
- How to upload the correct template or change context.

After identity validation, import review should show:

- Total rows.
- Ready rows.
- Rows needing review.
- Errors.
- Warnings.
- Suggested fixes.

Every error must identify row, field, problem, expected value, and suggested fix. Import validation must reuse confirmed Product and Specification Template rules; workbook validation is assistance, not a replacement for server/domain validation.

## Examples

Inside `Computers -> Laptops -> Gaming -> Lenovo`, Department, Category, Device Class, and Brand are known. The template excludes those editable columns and includes unresolved values such as Product Model, Product Name, Price, Currency, Condition, Availability, Quantity, CPU, RAM, Storage, GPU, Screen Size, and Operating System.

From the Laptops page, the template may include Device Class, Brand, Product Model, Product fields, and resolved Specification fields.

From Gaming Laptops, it may include Brand, Product Model, Product fields, and Specification fields.

From Lenovo Gaming Laptops, it may include Product Model, Product fields, and Specification fields.

Example preview:

```text
Included: Product Model, CPU, RAM, GPU, Price
Excluded: Brand — already known from context
Excluded: Category — already known from context
Excluded: Device Class — already known from context
```

Example mismatch:

```text
Template context:
Computers -> Laptops -> Gaming -> Lenovo

Current import context:
Computers -> Laptops -> Business -> HP

Upload the correct template or change the current context.
```

## Future Uses

The same Template Generation Engine may support:

- Excel Import.
- Excel Export.
- Bulk Update.
- CSV Import.
- Supplier Import.
- Future API Import.
- Future marketplace exchange.
- Future AI-assisted mapping.

All consumers must reuse the same context resolution and Template Definition. AI may suggest mappings for employee review but must not override template identity, confirmed field definitions, compatibility, or validation.

## العربية

## العنوان

محرك توليد القوالب الديناميكي الواعي بالسياق

## الحالة

معتمد؛ التنفيذ مخطط له في مهمة مستقبلية.

## السياق

يجب أن يكون إدخال المنتجات بالدفعات سريعا دون إضعاف جودة بيانات المنتجات. إن صيانة ملفات Excel ثابتة يدويا لكل تصنيف أو فئة جهاز أو علامة تجارية أو نموذج منتج ستكرر قواعد الكتالوج، وستصبح قديمة عند تغير قوالب المواصفات، وستطلب من الموظفين تكرار معلومات معروفة من موقعهم الحالي في الكتالوج.

يحدد QSC مواصفات المنتج بالفعل من التصنيف وفئة الجهاز الاختيارية، ويدعم إدخال المنتج الواعي بالسياق. لذلك يجب أن يوسع توليد القوالب هذه القرارات، لا أن ينشئ مصدرا منفصلا لقواعد المواصفات أو التحقق.

## القرار

يجب أن يولد QSC قوالب الاستيراد والتصدير والعمليات المجمعة ديناميكيا من سياق الإدخال الحالي. ويجب ألا تصان القوالب كملفات ثابتة لكل تركيبة في الكتالوج.

قد يحتوي سياق الإدخال على:

- `companyId`.
- `workspaceId`.
- `departmentId`.
- `categoryId`.
- `deviceClassId`.
- `brandId`.
- `productModelId`.
- `entryMethod`.
- `language`.
- الصلاحيات المستقبلية.

المسار المعتمد هو:

```text
سياق الكتالوج الحالي
-> تحديد قالب المواصفات
-> اكتشاف القيم المعروفة
-> حساب الأعمدة المطلوبة
-> توليد القالب الديناميكي
-> التنزيل / الاستيراد / التصدير / التحديث المجمع
```

تظهر القيم المعروفة في هوية القالب وتعليماته، لكنها لا تتكرر كأعمدة منتجات قابلة للتعديل. ويترك السياق الأعلى قيما أكثر دون تحديد، ولذلك ينتج أعمدة أكثر.

## الأسباب

- تقليل الإدخال المتكرر من الموظف.
- منع عدم تطابق التصنيف وفئة الجهاز والعلامة التجارية وبيانات المنتج.
- إبقاء الأعمدة المولدة متزامنة مع إعدادات الكتالوج المؤكدة.
- دعم الإدخال السريع بالدفعات دون تجاوز التحقق.
- تجنب صيانة ملفات ثابتة كثيرة تصبح قديمة.
- إعادة استخدام محرك واحد في الاستيراد والتصدير والعمليات المجمعة.
- الحفاظ على سياق المستأجر ومساحة العمل واللغة والصلاحيات المستقبلية.

## النتائج

- يتطلب توليد القالب سياق إدخال صريحا ومؤكدا.
- يستمر التصنيف وفئة الجهاز الاختيارية في تحديد قالب المواصفات وفقا لـ ADR-0002.
- ينتج السياق الأعمق أعمدة قابلة للتعديل أقل، وينتج السياق الأعلى أعمدة سياق مطلوبة أكثر.
- تعكس القوالب الجديدة تغييرات قالب المواصفات تلقائيا.
- قد تتطلب القوالب المولدة سابقا تحقق الإصدار والسياق عند الاستيراد.
- يجب أن تبقى الحقول المقدمة من السياق ظاهرة في بيانات الهوية والمعاينة حتى عند استبعادها من صفوف المنتجات.
- يتطلب توليد القالب وتنسيق المصنف والتخزين والتنزيل والرفع والتحليل تصميما تنفيذيا منفصلا.
- لا ينفذ هذا السجل Excel أو CSV أو الاستيراد أو التصدير أو التحديث المجمع.

## الاتجاه المعماري

محرك توليد القوالب قدرة لخدمة التطبيق والمجال، وليس منطق عمل داخل الواجهة أو خاصا بـ Excel.

```text
واجهة إدخال المنتج / التصدير / العمليات المجمعة
-> خدمة توليد القوالب
-> تحديد سياق الكتالوج
-> خدمة قالب المواصفات
-> متطلبات حقول المنتج
-> تعريف القالب
-> محول التنسيق (Excel / CSV / API مستقبلي)
```

تطلب المكونات المعاينات والملفات المولدة من خلال الخدمات. وتحسب الخدمات الحقول المعروفة والمطلوبة والاختيارية والمضمنة والمستبعدة. وتوفر المستودعات إعدادات الكتالوج المؤكدة. وتحول محولات التنسيق تعريف قالب مستقلا عن التنسيق إلى Excel أو CSV أو صيغ تبادل مستقبلية.

يجب أن تأتي أعمدة القالب فقط من:

- سياق الكتالوج المؤكد.
- قالب المواصفات المحدد.
- حقول قالب المواصفات.
- متطلبات حقول المنتج المؤكدة.

يجب ألا تكرر تعريفات المواصفات في كود خاص بـ Excel أو في الوثائق. ويجب ألا تنشئ حالات الاستخدام المختلفة منطق حساب قوالب منفصلا.

## هوية القالب

يجب أن يحدد كل قالب مولد مالكه وسياقه ولغته وإصدار تعريفه. ويجب أن يتضمن رأس ظاهر القيم المنطبقة من:

- القسم.
- التصنيف.
- فئة الجهاز.
- العلامة التجارية.
- نموذج المنتج.
- الشركة أو مساحة العمل.
- أنشئ بواسطة.
- تاريخ الإنشاء.
- إصدار القالب.
- معرف القالب.
- اللغة.

مثال:

```text
قالب منتجات QSC
القسم: الكمبيوترات
التصنيف: اللابتوبات
فئة الجهاز: جيمنج
العلامة التجارية: Lenovo
مساحة العمل: Quadcore
معرف القالب: TMP-LAP-GAM-LEN-001
إصدار القالب: 1.0
تاريخ الإنشاء: 2026-07-15
```

يجب أن تكون أسماء الملفات ذات معنى، مثل:

```text
QSC_Computers_Laptops_Gaming_Lenovo_Template_v1.xlsx
QSC_Laptops_Business_Dell_2026-07-15.xlsx
```

ولا يسمح بأسماء عامة مثل `Template.xlsx`.

## قواعد القالب المعرّف بذاته

يجب أن يكون كل قالب مولد مفهوما دون وثائق خارجية. وينبغي أن يحتوي المصنف المستقبلي على:

1. ورقة المنتجات.
2. ورقة التعليمات.
3. ورقة بيانات مرجعية اختيارية.

يجب أن تشرح ورقة التعليمات:

- غرض القالب.
- السياق الحالي.
- الحقول المطلوبة والاختيارية.
- الحقول المقدمة من السياق والمستبعدة.
- القيم المسموحة.
- الأمثلة.
- الأخطاء الشائعة.
- تعليمات الاستيراد.
- إصدار القالب.

يجب التمييز بوضوح بين الحقول المطلوبة والاختيارية والمقدمة من السياق والمستبعدة. وتأتي قواعد المطلوب والاختياري من قالب المواصفات المحدد ومتطلبات المنتج.

قد تشمل الميزات الذكية المستقبلية مؤشرات الحقول المطلوبة والقوائم المنسدلة وقواعد القيم المسموحة والتحقق من الأرقام والقيم المنطقية والقيم النموذجية ووصف الحقول وتعدد اللغات ومعاينة مناسبة للجوال.

قبل التنزيل، يجب أن تعرض المعاينة السياق الحالي والأعمدة المضمنة والمستبعدة والحقول المطلوبة والاختيارية وسبب كل استبعاد.

## التحقق عند الاستيراد

عند الرفع، يجب أن يقرأ QSC معرف القالب وإصداره والقسم والتصنيف وفئة الجهاز والعلامة التجارية ومساحة العمل ويتحقق منها عندما تنطبق. ويجب أن يقارن هوية القالب بسياق الاستيراد الحالي، وألا يتابع بصمت عند عدم التطابق.

يجب أن تشرح رسالة عدم التطابق:

- السياق الذي ينتمي إليه الملف.
- سياق الاستيراد الحالي للموظف.
- طريقة رفع القالب الصحيح أو تغيير السياق.

بعد التحقق من الهوية، يجب أن تعرض مراجعة الاستيراد:

- إجمالي الصفوف.
- الصفوف الجاهزة.
- الصفوف التي تحتاج إلى مراجعة.
- الأخطاء.
- التحذيرات.
- الإصلاحات المقترحة.

يجب أن يحدد كل خطأ الصف والحقل والمشكلة والقيمة المتوقعة والإصلاح المقترح. ويجب أن يعيد تحقق الاستيراد استخدام قواعد المنتج وقالب المواصفات المؤكدة؛ فالتحقق داخل المصنف مساعدة، وليس بديلا عن تحقق المجال أو الخادم.

## الأمثلة

داخل `الكمبيوترات -> اللابتوبات -> جيمنج -> Lenovo` تكون قيم القسم والتصنيف وفئة الجهاز والعلامة التجارية معروفة. يستبعد القالب هذه الأعمدة القابلة للتعديل، ويتضمن القيم غير المحددة مثل نموذج المنتج واسم المنتج والسعر والعملة والحالة والتوفر والكمية والمعالج والذاكرة والتخزين وكرت الشاشة وحجم الشاشة ونظام التشغيل.

من صفحة اللابتوبات، قد يتضمن القالب فئة الجهاز والعلامة التجارية ونموذج المنتج وحقول المنتج وحقول المواصفات المحددة.

ومن لابتوبات الألعاب، قد يتضمن العلامة التجارية ونموذج المنتج وحقول المنتج وحقول المواصفات.

ومن لابتوبات Lenovo للألعاب، قد يتضمن نموذج المنتج وحقول المنتج وحقول المواصفات.

مثال المعاينة:

```text
مضمن: نموذج المنتج، المعالج، الذاكرة، كرت الشاشة، السعر
مستبعد: العلامة التجارية — معروفة من السياق
مستبعد: التصنيف — معروف من السياق
مستبعد: فئة الجهاز — معروفة من السياق
```

مثال عدم التطابق:

```text
سياق القالب:
الكمبيوترات -> اللابتوبات -> جيمنج -> Lenovo

سياق الاستيراد الحالي:
الكمبيوترات -> اللابتوبات -> أعمال -> HP

ارفع القالب الصحيح أو غير السياق الحالي.
```

## الاستخدامات المستقبلية

قد يدعم محرك توليد القوالب نفسه:

- استيراد Excel.
- تصدير Excel.
- التحديث المجمع.
- استيراد CSV.
- استيراد الموردين.
- استيراد API مستقبلي.
- تبادل السوق المستقبلي.
- الربط بمساعدة الذكاء الاصطناعي مستقبلا.

يجب أن يعيد كل مستهلك استخدام تحديد السياق وتعريف القالب نفسيهما. ويمكن للذكاء الاصطناعي اقتراح ربط الحقول لمراجعة الموظف، لكنه لا يتجاوز هوية القالب أو تعريفات الحقول المؤكدة أو التوافق أو التحقق.
